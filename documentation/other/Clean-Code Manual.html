Über dieses Dokument

Dieses Dokument listet die in PROG2/PM2 einzuhaltenden Clean-Code Regeln. Es ist eine überarbeitete Fassung des in PM1/PROG1 verwendeten Materials (entsprechend markiert). Diese Clean-Code Regeln sind für das ganze Studium an der ZHAW gültig und werden, sofern nicht anders vermerkt, konsequent eingefordert. Die vorhandenen Beispiele sollen die Clean-Code-Regeln illustrieren. Sie dienen nicht dazu, neuen Stoff über den in der Vorlesung behandelten einzuführen.

Neue Elemente im Vergleich zu PM1 wurden mit Neu! markiert (über 20 Einträge).

Bestimmte Elemente werden nur in bestimmten Projekten (bezeichnet mit L1, L2, L3 und L4) bewertet.

Review-Kriterien im Detail

Lesbarkeit

Java-Namenskonventionen werden eingehalten (L1)

  • Klassennamen sind Upper Camel Case (MySpecialClass)

  • Methodennamen, Variablennamen und Feldnamen sind Lower Camel Case (someMethod())

  • Konstanten sind Grossbuchstaben (ABSOLUTE_NULL)

Java-Klasse mit eingehaltenen Namenskonventionen 
public class Temperature {

    private static final BigDecimal KELVIN_CELSIUS_DIFF = BigDecimal.valueOf(273.15);

    private final BigDecimal value;

    public Temperature(BigDecimal degreesKelvin) {
        this.value = degreesKelvin;
    }

    public BigDecimal toCelsius() {
        return value.add(KELVIN_CELSIUS_DIFF);
    }
}

Code ist aufgeräumt (L1)

Kein toter Code ist vorhanden

Entfernen Sie auskommentierten Code umgehend. Er reduziert die Lesbarkeit des Codes und wird spätestens nach der nächsten Umbenennung einer Variablen nutzlos. Falls Sie fürchten, die alte Implementierung noch einmal zu brauchen: Dafür ist die Versionskontrolle da.

Klasse mit totem Code 
public class Temperature {
    public static BigDecimal toCelsius(BigDecimal degreesKelvin) {
        // BigDecimal.valueOf(value.floatValue() + 273.15) (1)
        return degreesKelvin.add(BigDecimal.valueOf(273.15));
    }
}
1 Toter Code. Ist value dasselbe wie degreesKelvin?

Redundanter Kommentar

Importieren Sie nur, was Sie brauchen

Importieren Sie nur die Namen, die Sie in der aktuellen Datei tatsächlich brauchen. Unnötige Imports (und dazu gehören auch die sogenannten Star Imports wie import java.util.*;) sorgen zwar nur für einen kleinen Performance-Verlust und das auch nur beim Kompilieren, können Ihnen aber Wartungsprobleme bereiten. Folgender Code lief in Java 1.1 problemlos. Beim Upgrade auf Java 1.2 versagt er, auch wenn der Code nicht angefasst wurde. Warum?

Drohendes Unglück 
import java.math.BigDecimal;
import java.util.*;
import java.awt.*;

public class Test {

    public static void main(String[] args) {
        List list = new List();
    }
}

Der Typ List ist java.awt.List, in Java 1.1 der einzig verfügbare Listen-Typ. Mit Java 1.2 kommt mit java.util.List plötzlich ein zweiter Typ List dazu, womit der Code mehrdeutig wird.

Verzichten Sie auf Copy & Paste

Bevor Sie Code kopieren und an einer anderen Stelle einfügen, überlegen Sie sich, ob Sie nicht eine Methode schreiben und diese referenzieren können.

Besonders geeignete Kandidaten:

  • Berechnungen bzw. Zusammensetzung von Zeichenketten (z.B. Vor- und Nachname)

  • Objekte erstellen aus SQL-ResultSet und umgekehrt

  • Hilfsmethoden, z.B. zum Ausgeben eines Datums

Ziehen Sie Konstanten Magic Numbers vor

Magic Numbers sind (meist) Zahlen, die plötzlich im Code auftauchen — und niemand weiss, was sie bedeuten und warum gerade der angegebene Wert der richtige ist.

Temperaturkonversion mit Magic Number 
public class Temperature {
    public static BigDecimal toCelsius(BigDecimal degreesKelvin) {
        return degreesKelvin.add(BigDecimal.valueOf(273.15)); (1)
    }
}
1 273.15 ist eine Magic Number

Erklären Sie die Magic Numbers mit einem Kommentar oder — meist besser — durch eine Konstante, da Sie dann immer denselben Wert referenzieren und ihn so nur einmal zu ändern brauchen.

Magic Number als Konstante ausgedrückt 
public class Temperature {

    private static final BigDecimal KELVIN_CELSIUS_DIFF = BigDecimal.valueOf(273.15); (1)

    public static BigDecimal toCelsius(BigDecimal degreesKelvin) {
        return degreesKelvin.add(KELVIN_CELSIUS_DIFF);
    }
}
1 Benannte Konstante
Formatieren Sie Code konsistent

Ob Sie Tabs oder Leerschläge zum Einrücken des Codes verwenden, spielt keine grosse Rolle. Ebenso wenig, ob Sie Blöcke auf derselben Zeile mit if, for usw. beginnen oder in der Zeile darunter. Nur tun Sie es konsistent. Dies verbessert die Lesbarkeit und sorgt für minimalen "Lärm" durch Formatierungsänderungen beim Code Review. Erstellen Sie jeweils eigene commits für Formatierungänderung und für Logikänderung, bitte nicht mischen.

Verwenden Sie den Eclipse Formatter, um den Code automatisch formatieren zu lassen. Auf Windows drücken Sie dazu Ctrl + ⇧ + F und auf Mac ⌘ + ⇧ + F.

Sie müssen im ganzen Team dieselben Einstellungen bezüglich Tabulatoren und Leerzeichen haben. Wenn nach dem Einsatz des Formatters alle Zeilen als "verändert" angezeigt werden, müssen Sie zuerst diese Einstellungen kontrollieren.
Verzichten Sie auf veraltete Schnittstellen

Java existiert seit 1995. Seither sind etliche Technologien und Schnittstellen dazu gekommen, die teilweise vorhandene Schnittstellen ersetzen. Um die Rückwärtskompatibilität zu gewährleisten, wurden die Schnittstellen, die ersetzt wurden, oftmals nicht entfernt. Das heisst aber nicht, dass man Sie weiterhin verwenden soll. Achten Sie darauf, insbesondere folgende Schnittstellen nicht mehr zu verwenden:

  • java.util.Calendar (Ersatz: java.time)

Sinnvolle Namensgebung (L1)

Achten Sie bei der Benennung von Variablen, Klassen und Methoden darauf, möglichst beschreibende Namen zu verwenden. Dies hilft, Ihren Code nachzuvollziehen.

Unklare Variablen- und Methodennamen 
public class Temperature {
    public static BigDecimal calculate(BigDecimal value) { (1) (2)
        return value.add(273.15);
    }
}
1 Was macht calculate genau?
2 Was muss man bei value angeben? Celsius? Fahrenheit?
Methoden- und Variablennamen kommunizieren Zweck 
public class Temperature {
    public static BigDecimal toCelsius(BigDecimal degreesKelvin) { (1) (2)
        return degreesKelvin.add(273.15);
    }
}
1 Mit dem Methodennamen toCelsius wird kommuniziert, dass der übergebene Wert in Grad Celsius konvertiert wird.
2 Dank dem Variablennamen degreesKelvin weiss der Programmierer, dass die Gradangabe in Kelvin zu erfolgen hat.

Verzichten Sie ausserdem auf die Einbettung von Typen in Variablennamen, Klassen oder Interfaces ("ungarische Notation").

Ungarische Notation 
public class MathClass { (1)
    private static final double S_PI = 3.141; (2)

    public static double circleArea(double dRadius) { (3)
        return S_PI * dRadius * dRadius;
    }
}
1 Aufgrund des Suffix Class wissen Sie, dass es sich bei MathClass um eine Klasse handelt.
2 Das Prefix S_ kommuniziert, dass es sich um eine statische Variable handelt.
3 Das d signalisiert, dass in dRadius ein double gespeichert ist.

Methodengestaltung (L1,L3)

Methoden sind maximal 40 Zeilen lang (L1)

Beschränken Sie die Länge einer Methode auf maximal 40 Zeilen. Reichen die 40 Zeilen nicht aus für die Logik, die Sie unterbringen wollen, machen Sie nicht einfach die Zeilen länger. Lagern Sie stattdessen sinnvolle Teile der Logik (z.B. Bedingungen, Teilberechnungen) in weitere Methoden aus.

Lange Parameterlisten durch Builder oder Argumentobjekt ersetzen (L3)

Neu!

Vermeiden Sie lange Parameterlisten bei Ihren Methoden, da sie den Code schwer wart- und lesbar machen. Ausserdem wird der Code schwer zu benutzen, da Sie nicht mehr leicht ableiten können, welchen Effekt verschiedene Angaben haben. Beschränken Sie die Anzahl der Parameter auf 4 bis 5 (Faustregel).

Methode mit langer Argumentliste 
public class BookSearch {

    public List<Book> searchBooks(String title, String author, String isbn, int year,
                                  String publisher, Set<Category> categories) {
        // search catalogue
    }
}

Ersetzen Sie lange Parameterlisten durch ein Argumentobjekt.

Argumentobjekt für Suchkriterien ersetzt lange Parameterliste 
public class BookSearchCriteria {

    private String title;
    private String author;
    private String isbn;

    // etc.

    public String getTitle() {
        return this.title;
    }

    public void setTitle(String title) {
        this.title = title;
    }

    // etc.
}

public class BookSearch {

    public List<Book> searchBooks(BookSearchCriteria criteria) {
        // search catalogue
    }
}

Wenn Sie den Code noch ein wenig benutzerfreundlicher machen wollen, verwandeln Sie BookSearchCriteria in einen Builder.

Builder zur Erstellung von Argumentobjekt 
public class BookSearchCriteria {

    private final String title; (1)
    private final String author;

    private BookSearchCriteria(String title, String author) { (2)
        this.title = title;
        this.author = author;
    }

    public String getTitle() {
        return this.title;
    }

    public static class Builder {

        private String title; (3)
        private String author;

        public Builder setTitle(String title) {
            this.title = title;
            return this;
        }

        public Builder setAuthor(String author) {
            this.author = author;
            return this;
        }

        public BookSearchCriteria build() {
            return new BookSearchCriteria(this.title, this.author); (4)
        }
    }
}
1 Deklarieren Sie die Variablen final, damit die Suchkriterien nach Erstellung des Objekts unveränderlich sind.
2 Setzen Sie den Konstruktor private, damit der Builder nicht umgangen werden kann.
3 Wenn sinnvoll, setzen Sie hier Standardwerte für die Suchkriterien.
4 Validieren Sie bei Bedarf die Argumente vor Erstellung des Argumentobjekts.
Suchkriterien werden mit Builder erstellt 
BookSearchCriteria criteria = new BookSearchCriteria.Builder()
        .setAuthor("Joshua Bloch")
        .setTitle("Effective Java")
        .build();

Klassengestaltung

Klassen haben Verhalten (L1)

Achten Sie darauf, dass Ihre Klassen nicht nur Daten speichern, sondern auch Verhalten (d.h. Methoden jenseits von Getters und Setters) aufweisen. Denn das ist der Kern der objektorientierten Programmierung.

Klasse, die nur Daten speichert 
public class Circle {

    private final double radius;

    public Circle(double radius) {
        this.radius = radius;
    }

    public double getRadius() {
        return radius;
    }
}
Klasse mit Verhalten 
public class Circle {

    private final double radius;

    public Circle(double radius) {
        this.radius = radius;
    }

    public double getRadius() {
        return radius;
    }

    public double getArea() {
        return radius * radius * Math.PI;
    }

    public double getCircumference() {
        return 2 * Math.PI * radius;
    }
}

Klassenoberfläche ist minimal (L1)

Machen Sie jede Klasse, ihre Eigenschaften und Methoden so wenig zugänglich wie möglich. Dies bedeutet: Können Sie ein Feld oder eine Methode private deklarieren, deklarieren Sie sie private. Verwenden Sie weniger restriktive Zugangsbeschränkungen (protected oder gar public) nur, wenn es unbedingt nötig ist.

Instanzfelder sollten nur in Ausnahmefällen public sein. Generieren Sie auch nicht für jedes Feld einen Getter oder Setter, nur weil die IDE dies Ihnen anbietet.

All diese Massnahmen reduzieren die Abhängigkeiten zwischen den verschiedenen Klassen und machen die Wartung des Codes einfacher.

Klassen stehen für sich allein (L2)

Achten Sie darauf, dass Ihre Klassen für sich allein stehen. Symptome, dass Ihre Klassen nicht für sich alleine stehen:

  • Eine Klasse und ihre Methoden müssen andauernd andere Klassen nach ihrem Zustand fragen.

  • Eine Klasse und ihre Methoden müssen mehrere Methoden einer anderen Klasse aufrufen, um etwas zu erledigen.

  • Eine Klasse und ihre Methoden müssen Methoden einer anderen Klasse in einer bestimmten Reihenfolge aufrufen.

  • Eine Klasse und ihre Methoden müssen über mehrere Objekte hinweg nach Daten fragen (z.B. this.getOther().getAnother().getData())

In diesem Fall ist es angebracht, die Funktionalität in die anderen Klassen zu verschieben respektive von dort in die aktuelle Klasse zu bewegen, um die Abhängigkeiten zu reduzieren.

Sinnvolle Abstraktionsebene (L1)

Interfaces statt abstrakte Klassen

Ziehen Sie Interfaces abstrakten Klassen vor (Composition over Inheritance). Abstrakte Klassen geben ein enges Korsett vor: Abgeleitete Klassen müssen eine strikte Hierarchie bilden (Ist-ein-Beziehung) und das Liskovsche Substitutionsprinzip einhalten. Ausserdem kann in Java jeweils nur von einer Klasse geerbt werden. Interfaces sind im Vergleich viel flexibler: Existierende Klassen können jederzeit zusätzliche Interfaces implementieren und nichthierarchische Typsysteme bilden. Ausserdem können existierende Klassen nachträglich mit zusätzlicher Funktionalität (sogenannten Mixins) ausgerüstet werden, ohne die Klassen selber anpassen zu müssen.

Keine unnötige Abstraktion

Verzichten Sie darauf, auf Vorrat abstrakte Klassen und Methoden sowie Interfaces zu erstellen für den Fall, dass man sie in Zukunft vielleicht einmal brauchen könnte. Existiert für ein Interface oder eine abstrakte Klasse nur eine einzelne Implementierung, verzichten Sie darauf. Gleiches gilt für ungenutzte Methoden oder Klassen, die fast keine Aufgaben haben.

Testing

Ein sinnvoller Unit-Test pro Klasse ist vorhanden (L1)

Zu jeder Klasse in Ihrem Projekt sollte sich ein sinnvoller Unit Test befinden, der ein Mindestmass an Logik (= ein Ausdruck, der etwas berechnet) testet (= mindestens eine Assertion ist vorhanden).

Ein sinnvoller Test 
class EmployeeTest {

    @Test
    void fullNameComposedOfFirstAndLastName() {
        Employee employee = new Employee();
        employee.setName("Diana");
        employee.setLastName("Prince");
        assertEquals("Diana Prince", employee.getFullName()); (1)
    }
}
1 Mindestens eine Assertion, die mehr als assertTrue(true) testet, ist gefordert!

Logik mit Tests abgedeckt (L1)

Die wesentliche Logik Ihrer Software ist mit automatisierten Tests abgedeckt. "Wesentlich" bedeutet, dass Sie weder Getter noch Setter testen brauchen, solange sie nichts anderes machen, als Felder zu lesen beziehungsweise zu setzen. Dasselbe gilt für Hilfsmethoden wie toString(). Bei allem anderen sollte die korrekte Funktionsweise durch einen automatisierten Test mit geeigneten Assertions sichergestellt sein.

Selenium WebDriver ist ein nützliches Werkzeug für Integrationstests, mit dem Sie die Ausgaben Ihrer Web-Applikation testen können.

Mock-Objekte statt externe Kollaborateure (L2)

Komponenten (wie ein Data Access Object) verwenden in der Regel andere Komponenten (wie eine Datenbankverbindung), um eine Aufgabe zu erledigen. Um eine Komponente sinnvoll zu testen, ist es ratsam, diese von den anderen Komponenten zu isolieren. In Kollaborateure als Konstruktor-Argumente (L3) wird der erste Schritt erklärt, um eine Komponente zu isolieren, nämlich die Übergabe von Kollaborateuren als Konstruktorargument. Der zweite Schritt ist, anstelle des eigentlichen Kollaborateurs eine Testattrappe oder Mock-Objekt zu übergeben. Mit Hilfe einer Testattrappe können Sie in einem Test verifizieren, wie die zu testende Komponente mit dem eigentlichen Kollaborateur interagieren würde und ob sie es richtig tut.

Zu testen: Wird die Buchbewertung korrekt hinzugefügt?
class BookService {

    private final BookRepository bookRepository;

    public BookService(BookRepository bookRepository) {
        this.bookRepository = bookRepository;
    }

    public void reviewBook(ISBN isbn, Customer reviewer, String review) {
        Book book = this.bookRepository.getBook(isbn);
        book.addReview(reviewer, review);
    }
}

class BookRepository {

    private final DataSource dataSource;

    public BookRepository(DataSource dataSource) {
        this.dataSource = dataSource;
    }

    public Book getBook(ISBN isbn) {
        // call database, return book
    }
}

Wie lässt sich testen, ob die Bewertung zum richtigen Buch hinzugefügt würde, ohne die Datenbank zu bemühen? Indem man das BookRepository durch eine Testattrappe ersetzt, die ohne Datenbank auskommt.

Test des BookService in Isolation 
class BookServiceTest {

    @Test
    void reviewAdded() {
        Book testBook = new Book();
        TestBookRepository bookRepository = new TestBookRepository(testBook);
        BookService bookService = new BookService(bookRepository);

        bookService.reviewBook(new ISBN("123"), new Customer(), "Great book");

        assertEquals(bookRepository.getLastUsedIsbn(), new ISBN("123"));
        assertEquals(1, testBook.getReviews().size());
    }

    class TestBookRepository extends BookRepository { (1)

        private final Book book;
        private ISBN lastUsedIsbn;

        TestBookRepository(Book book) { (2)
            this.book = book;
        }

        @Override
        public Book getBook(ISBN isbn) { (3)
            this.lastUsedIsbn = isbn; (4)
            return book;
        }

        ISBN getLastUsedIsbn() {
            return lastUsedIsbn;
        }
    }
}
1 Indem TestBookRepository von BookRepository erbt, kann es anstelle des BookRepository, das eine Datenbankverbindung bräuchte, an den BookService übergeben werden.
2 Die Methode Book getBook(ISBN) muss ein Buch zurückgeben. Dieses wird über den Konstruktor des TestBookRepository für die spätere Rückgabe bereitgelegt.
3 Durch das Überschreiben der Methode Book getBook(ISBN) wird die Datenbankverbindung der ursprünglichen Implementierung nicht mehr benötigt. Die restliche Logik in der ursprünglichen Implementierung kann ebenfalls ignoriert werden, da man sich nur noch für die Interaktion von BookService mit BookRepository interessiert: Wird die richtige ISBN übergeben und das Buch korrekt weiterverwendet?
4 Damit man überprüfen kann, ob die richtige ISBN übergeben wurde, muss das TestBookRepository sie sich merken und über einen separaten Getter zugänglich machen.
Die ist eine Vorarbeit für die Benutzung von Mock-Libraries. Sie vereinfachen das Schreiben von komplexen Tests.

Kollaborateure als Konstruktor-Argumente (L3)

Wenn Ihre Klasse (nennen wir sie A) eine andere Klasse (B) verwendet, um eine Aufgabe zu erledigen, ist B ein Kollaborateur oder eine Abhängigkeit (Dependency) von A. Übergeben Sie B immer als Konstruktor-Argument an A, statt B im Konstruktor von A mit new zu erstellen. Das macht es einfacher, die Klasse zu testen.

Abhängigkeiten werden im Konstruktor erstellt 
public class SomeEntityService {

    private final SomeEntityDao someEntityDao;

    public SomeEntityService() {
        this.someEntityDao = new SomeEntityDao();
    }
}

Wenn Sie in einem Test SomeEntityDao durch einen Mock ersetzen wollen, sind Sie aufgeschmissen. Übergeben Sie stattdessen SomeEntityDao als Konstruktorargument und Sie können im Rahmen eines Tests eine andere Instanz übergeben.

Abhängigkeiten werden als Konstruktorargument übergeben 
public class SomeEntityService {

    private final SomeEntityDao someEntityDao;

    public SomeEntityService(SomeEntityDao dao) { (1)
        this.someEntityDao = dao;
    }
}
1 Potentielle Verbesserung: Extrahieren Sie ein Interface aus SomeEntityDao, dann brauchen Sie nicht einmal eine Mocking-Bibliothek, um einen Mock zu erstellen.

Unfreundlich zum Testen sind auch statische Methoden, da Sie diese ebenfalls nicht ersetzen können:

Statische Methoden sind untestbar 
public class SomeEntityService {

    private final SomeEntityDao someEntityDao;

    public SomeEntityService(SomeEntityDao dao) {
        this.someEntityDao = dao;
    }

    public void createSomeEntity(String name) {
        entity = new SomeEntity();
        entity.setName(name);
        entity.dateCreated(Instant.now()) (1)
        this.someEntityDao.save(entity);
    }
}
1 In einem Test wissen Sie nicht, welche Zeit gesetzt wird. Sie können sich in Assertions bestenfalls annähern und einen Hack wie assertTrue(dateCreated.isAfter(Instant.now().minusSeconds(2))) in Kombination mit assertTrue(dateCreated.isBefore(Instant.now())) verwenden.

Abhilfe schafft, keine statischen Methoden zu verwenden oder sie mit Argumenten zu modifizieren. java.time bietet eine elegante Möglichkeit mit der abstrakten Clock, für die verschiedene Implementierungen existieren — unter anderem die Systemuhr, die jeweils die aktuelle Zeit zurückgibt, und eine Variante, die immer dieselbe vorgegebene Zeit zurückgibt (Clock.fixed(Instant.now(), ZoneOffset.UTC)) und damit ideal für Tests ist.

Parameterisierte statische Methode erhöht Testbarkeit 
public class SomeEntityService {

    private final SomeEntityDao someEntityDao;

    public SomeEntityService(SomeEntityDao dao) { (1)
        this.someEntityDao = dao;
    }

    public void createSomeEntity(String name, Clock clock) {
        entity = new SomeEntity();
        entity.setName(name);
        entity.dateCreated(Instant.now(clock)) (2)
        this.someEntityDao.save(entity);
    }
}
1 Übergeben Sie Clock als Konstruktorargument. Dies erlaubt Ihnen später, die Uhr für Tests auszutauschen.
2 Übergeben Sie die Clock zur Bestimmung der aktuellen Zeit.

Ausnahmen werden behandelt (L2)

Neu!

Treffen Sie in Ihrem Programm auf eine Ausnahmesituation, werfen Sie eine Exception, anstatt das Problem zu ignorieren und auf das Beste zu hoffen.

Unerwartete Situation löst Exception aus 
File someFile = new File(path);
if (!someFile.canRead()) {
    String message = String.format("File '%s' is not readable", path); (1)
    throw new FileNotReadableException(message);
}
// Read file
1 Diagnoseinformation, die mitteilt, welche Datei nicht lesbar ist.

Um eine eigene Exception zu definieren, erben Sie von RuntimeException und überschreiben Sie die Konstruktoren. Achten Sie in jedem Fall darauf, Diagnoseinformationen mitzugeben. Das erleichtert Ihnen die Fehlerdiagnose und Behebung.

Selbst definierte Exception 
public class FileNotReadableException extends RuntimeException {
    public FileNotReadableException(String message) { (1)
        super(message);
    }

    public FileNotReadableException(String message, Throwable cause) { (2)
        super(message, cause);
    }
}
1 Konstruktor, der eine einzelne Mitteilung akzeptiert.
2 Konstruktor, dem Sie zusätzlich zur Mitteilung eine existierende Exception mitgeben können, die die Ursache Ihrer Exception ist (siehe unten).
Das JDK definiert einige Exceptions wie IllegalArgumentException. Verwenden Sie diese, wenn sie passen.

Wenn Sie JDK APIs oder Bibliotheken nutzen, kann es sein, dass diese Exceptions werfen. Behandeln Sie diese angemessen:

  • Wenn möglich und sinnvoll (was selten der Fall ist), versuchen Sie den Fehler in der Applikation zu korrigieren. Bei einem fehlgeschlagenen Internet-Zugriff können Sie zum Beispiel den Zugriff erneut versuchen.

  • Konvertieren Sie die zugrunde liegende Exception in eine geeignete Exception und werfen Sie diese.

  • Schreiben Sie eine Meldung ins Log (siehe Logging anstelle von System.out (L3)).

Ignorierte Exception 
byte[] buffer = new byte[1024];
try (FileInputStream stream = new FileInputStream("/some/file.txt")) {
    stream.read(buffer);
} catch (IOException e) {
    // ignore exception
}
// do something with the buffer's contents

Wenn Exceptions ignoriert werden, läuft das Programm weiter und kommt dadurch potentiell in einen inkonsistenten Zustand. In diesem Beispiel weiss man unter anderem nicht, was in der Variable buffer gespeichert ist. Zudem erfährt man während des Betriebs nicht, dass es ein Problem gegeben hat.

Exception wird konvertiert und erneut geworfen 
byte[] buffer = new byte[1024];
try (FileInputStream stream = new FileInputStream("/some/file.txt")) {
    stream.read(buffer);
} catch (IOException e) {
    throw new FileReaderException("Could not read file", e); (1)
}
// do something with the buffer's contents
1 Konvertiert die IOException in eine RuntimeException und wirft diese erneut.
Wenn Sie eine Exception konvertieren, geben Sie sie an die nächst höhere Exception (in diesem Fall FileReaderException) weiter und schlucken Sie sie nicht. Ansonsten geht der Stacktrace verloren und Sie wissen nicht, was die eigentliche Ursache der FileReaderException ist.

Logging anstelle von System.out (L3)

Neu!

Geben Sie Debug- und Informationsmeldungen, Warnungen und Fehler mittels Logging aus anstatt System.out zu verwenden. Logging ermöglicht, während der Entwicklung und in der Produktion das Log Level einzustellen. Sie können damit regeln, wie viele Informationen ausgegeben werden sollen. Zudem können Sie steuern, ob die Ausgaben auf die Konsole oder zum Beispiel in eine Datei geschrieben werden sollen. All das geht mit System.out respektive System.err nicht.

Exception wird direkt auf die Konsole geschrieben 
public EmployeeService(ServletContext servletContext) {
    try {
        System.out.println("Establishing database connection"); (1)
        loadH2Driver();
        establishH2DBConnection(servletContext);
        setUpH2Database();
    } catch (Exception e) {
        e.printStackTrace(); (2)
    }
}
1 Gibt Debug-Meldung auf der Konsole aus.
2 Gibt Stacktrace auf der Konsole aus.

Es gibt viele Möglichkeiten, um in Java zu loggen. Java bringt im Package java.util.logging ("JUL" genannt) sogar selber die nötigen Werkzeuge mit.

Exception wird geloggt 
public class EmployeeService {

    private static final Logger LOGGER = Logger.getLogger(EmployeeService.class.getName()); (1)

    public EmployeeService(ServletContext servletContext) {
        try {
            LOGGER.info("Establishing database connection"); (2)
            loadH2Driver();
            establishH2DBConnection(servletContext);
            setUpH2Database();
        } catch (Exception e) {
            LOGGER.log(Level.SEVERE, "Could not establish database connection", e); (3)
        }
    }
}
1 Initialisiert den Logger. Achten Sie darauf, den Namen der umgebenden Klasse (EmployeeService in diesem Fall) zu verwenden und das Feld static final zu deklarieren.
2 Gibt Debug-Meldung via Logger aus, wenn das Log Level INFO oder tiefer ist.
3 Gibt Fehlermeldung via Logger aus, wenn das Log Level SEVERE oder tiefer ist.

Wenn Sie Log-Meldungen aus Variablen zusammenbauen, verwenden Sie keine String-Konkatenation.

String-Konkatenation für Log-Meldungen 
LOGGER.info("Unknown option " + userOption + ", using default " + defaultOption);

Die String-Konkatenation wird immer ausgeführt, selbst wenn die Log-Nachricht gar nicht ausgegeben wird, weil das Log-Level der Meldung im konfigurierten Log-Level der Applikation nicht eingeschlossen ist (z.B. bei error wird info nicht ausgegeben). Nutzen Sie daher Variablen-Interpolation. Die Variablen werden dabei erst eingesetzt, wenn die Nachricht tatsächlich ausgegeben wird.

Variablen-Interpolation für Log-Meldungen 
LOGGER.log(Level.INFO, "Unknown option {0}, using default {1}", userOption, defaultOption);
Was durch die Logger wo (Konsole, Datei, …) ausgegeben wird, können Sie über eine Datei namens logging.properties steuern. Wie Sie sie nutzen, wird zum Beispiel im Buch Java SE 8 Standard-Bibliothek von Christian Ullenboom (Rheinwerk) im Kapitel Logging mit Java beschrieben.

Vernünftiger Einsatz von Feldern (L1)

Speichern Sie nur in Feldern, was sie für einen längeren Zeitraum (= länger als der aktuelle Methodenaufruf) brauchen. Für alles andere reicht eine lokale Variable.

Interfaces statt konkrete Typen (L1)

Beziehen Sie sich beim Referenzieren von Parametern, Rückgabewerten, Variablen und Feldern auf Interfaces (wie List) anstatt auf konkrete Typen (wie ArrayList), wenn ein geeignetes Interface existiert. Verwenden Sie konkrete Typen nur beim Erstellen des Typs. Dies macht Ihr Programm viel flexibler. Wenn Sie feststellen, dass eine andere Implementierung eines Interfaces geeigneter ist, brauchen Sie nur den Konstruktoraufruf auszutauschen und Ihr Programm funktioniert weiter.

Interface wird anstelle des konkreten Typs verwendet 
SortedSet<String> aSet = new TreeSet<String>();

Wenn Sie mehrere Interfaces zur Auswahl haben, verwenden Sie dasjenige, das Ihnen die Funktionalität bietet, die Sie benötigen.

Achtung vor versteckten Anforderungen

Achten Sie darauf, das Interface spezifisch genug zu wählen, um nicht über versteckte Anforderungen zu stolpern. Folgende Deklaration könnte gefährlich werden:

Set<String> aSet = new TreeSet<String>();

Wenn Sie ein TreeSet verwenden, werden die im Set gespeicherten Strings automatisch sortiert, selbst wenn Sie das Set nur über das Interface Set ansprechen. Wenn Sie eines Tages das TreeSet aus Performance-Gründen in ein HashSet verwandeln, geht die Eigenschaft der automatischen Sortierung verloren. Wenn es für das restliche Programm wichtig ist, dass die Werte sortiert sind, wählen Sie als Interface SortedSet, da dieses die Sortierung garantiert. Sonst verzichten Sie am besten gleich auf TreeSet und wählen von Beginn weg HashSet.

Beschreiben Sie in Javadoc den Zweck der von Ihnen erstellten Klassen und Interfaces.

Ressourcen werden geschlossen (L2)

Neu!

Schliessen Sie alle Ressourcen (Datenbankverbindungen, Dateien, Streams, …), die Sie geöffnet haben, sobald Sie sie nicht mehr brauchen. Wenn Sie nicht sicher sind, ob etwas geschlossen werden muss, schauen Sie ins Javadoc des entsprechenden Typs: Wird Closeable oder AutoCloseable implementiert, handelt es sich um eine schliessbare Ressource und es gibt die Methode close().

Wenn Sie eine Ressource vergessen zu schliessen, kann ein Ressourcen-Leck auftreten. Dies bedeutet, dass Ihr Programm die Ressource besetzt hält (was Speicher und weitere Ressourcen kosten kann, z.B. File Handles) und im schlimmsten Fall andere Programme (oder sogar Ihr eigenes) nicht weiter auf die Ressource zugreifen können oder dass Ihr Computer den Dienst versagt.

Verwenden Sie nach Möglichkeit Try-With-Resources, weil dann Java für Sie darauf achtet, die Ressource korrekt zu schliessen.

Ressource wird automatisch geschlossen dank Try-With-Resources 
try (DirectoryStream<Path> directoryStream =
Files.newDirectoryStream(LOG_FOLDER, FILE_FILTER)) { (1)
    // do something with the directoryStream
} (2)
1 newDirectoryStream() erstellt ein directoryStream für Sie, das Sie nach der Verwendung schliessen müssen.
2 Weil das directoryStream in der Klammer nach try geöffnet wird, schliesst es Java automatisch für Sie, sobald der try-Block beendet wird.

Damit Sie Try-With-Resources verwenden können, muss die Ressource das Interface AutoCloseable implementieren. Tut sie es nicht, müssen Sie die Ressource selber korrekt schliessen.

Ressource wird von Hand geschlossen 
DirectoryStream<Path> directoryStream = null; (1)
try {
    directoryStream =
        Files.newDirectoryStream(LOG_FOLDER, FILE_FILTER);
    // do something with the directoryStream
} catch (IOException e) {
    throw new RuntimeException(e);
} finally { (2)
    if (directoryStream != null) { (3)
        try {
            directoryStream.close();
        } catch (IOException e) { (4)
            throw new RuntimeException(e);
        }
    }
}
1 Sie müssen die Ressource ausserhalb des try-Blocks deklarieren, damit Sie in finally darauf Zugriff haben.
2 Die Ressource muss immer innerhalb von finally geschlossen werden, nicht innerhalb von try. Nur so wird sichergestellt, dass wenn im try-Block eine Exception auftritt, die Ressource dennoch geschlossen wird.
3 Weil Sie nicht wissen, ob directoryStream bereits initialisiert wurde, müssen Sie auf null prüfen.
4 Weil close() selber eine Exception werfen kann, dürfen Sie sich über einen weiteren try-Block freuen.

Wenn Sie häufiger Try-With-Resources nicht verwenden können, tun Sie sich den Gefallen und führen Sie die Hilfsmethode closeQuietly(Closeable) ein:

Hilfsmethode closeQuietly(Closeable) 
public final class Util {

    private Util() {
        // no instances
    }

    public static void closeQuietly(Closeable closeable) {
        if (closeable == null) {
            return;
        }

        try {

Überarbeitet!
Argumentüberprüfung

Überprüfen Sie Argumente, bevor Sie sie weiterverarbeiten. Zum Beispiel dürfen Index-Angaben für Arrays nicht negativ sein oder eine Division durch Null ist nicht erlaubt. Indem Sie die Argumente frühzeitig validieren und Exceptions werfen, können Sie Ihren Code robuster gestalten, Sicherheitsprobleme und Exceptions verhindern.

Object-Klassenvertrag wird eingehalten (L1)

Vermeiden Sie nach Möglichkeit, die Methoden equals(Object) und hashCode() der Klasse Object zu überschreiben. Falls Sie es doch tun, beachten Sie folgende Kriterien:

  • Die Klasse muss ein Value Type, also unveränderlich, sein.

  • Wenn equals(Object) überschrieben wird, muss gleichzeitig auch hashCode() überschrieben werden und umgekehrt.

  • Der Klassenvertrag, also die Javadoc-Beschreibung, muss unbedingt eingehalten werden.

Halten Sie sich unbedingt an diese Regeln, da es ansonsten zu überraschenden Fehlern kommen kann, beispielsweise dass Objekte in sortierten Collections nicht am richtigen Ort erscheinen oder nicht gefunden werden. Wie Sie die Anforderungen richtig umsetzen, ist detailliert im Buch Effective Java beschrieben (siehe Referenzen).

Streams sind seiteneffektfrei (L3)

Neu!

Das Java Streams API (nicht zu verwechseln mit OutputStream und InputStream für I/O) ermöglicht, die deklarative Verarbeitung von Daten, wie das in funktionalen Programmiersprachen und SQL der Fall ist. Ein nützlicher Nebeneffekt des Stream API ist, dass die Parallelisierung der Datenverarbeitung sehr einfach möglich und damit Ihr Programm einfach schneller gemacht werden kann. Um all diese Möglichkeiten nutzen zu können, dürfen Sie nicht die klassische objektorientierte (mit Seiteneffekten) mit der funktionalen Programmierung (ohne Seiteneffekte) vermischen.

Folgender Code erstellt eine Liste, welcher Vorname wie häufig vorkommt. Er verwendet einerseits das Stream API (forEach()), nutzt aber gleichzeitig das herkömmliche objektorientierte API.

Seiteneffektbehaftete Nutzung von Streams 
List<Person> persons = new ArrayList<>();

Map<String, Integer> occurrences = new HashMap<>();
persons.forEach(person ->
        occurrences.merge(person.getFirstName(), 1, Integer::sum)); (1)
1 Aus dem Stream heraus wird die Map occurrences modifiziert, die ausserhalb des Stream existiert. Dies ist ein klassischer Seiteneffekt.

Entscheiden Sie sich in diesem Fall entweder für die klassische objektorientierte Variante mit for-Schleife oder nutzen Sie das Stream API richtig:

Seiteneffektfreie Nutzung von Streams 
List<Person> persons = new ArrayList<>();

Map<String, Long> occurrences = persons.stream()
        .collect(groupingBy(Person::getFirstName, counting())); (1)
1 Anstatt occurrences aus dem Stream heraus zu modifizieren, liefert der Stream das Endresultat. Dies ist seiteneffektfrei.

Standard-Java-Verzeichnisstruktur (L2)

Neu!

Legen Sie Ihre Klassen, Tests und Ressourcen (wie Konfigurationsdateien) in der Standard-Java-Verzeichnisstruktur ab:

src/main/java

Platz für Java-Klassen, jeweils in einem Unterverzeichnis korrespondierend zum Package-Namen (die öffentliche Klasse Test im Package com.example kommt nach src/main/java/com/example/Test.java). Speichern Sie jede Klasse in einer eigenen Datei, egal ob sie öffentlich ist oder nicht.

src/main/resources

Speichern Sie hier Konfigurationsdateien oder ähnliches, das nicht zur webapp gehört.

src/main/webapp

Legen Sie in diesem Verzeichnis Ihre Webapplikation ab.

src/test/java

Platz für Test-Klassen. Der Test für die öffentliche Klasse Example im Package com.example kommt nach src/test/java/com/example/ExampleTest.java. Es ist wichtig, dass Tests in demselben Package liegen wie die zu testenden Klassen, damit Sie in den Tests auf Methoden und Felder zugreifen können, die package private sind.

src/test/resources

Speichern Sie hier Konfigurationsdateien oder ähnliches für Tests.

Vernünftige Objektlebenszeiten (L3)

Neu!

Speichern Sie nur in Feldern, was sie für einen längeren Zeitraum (= länger als der aktuelle Methodenaufruf) brauchen. Für alles andere reicht eine lokale Variable.

Achten Sie insbesondere darauf, aufwendig zu erstellende Objekte wiederzuverwenden und nicht unnötigerweise neu zu kreieren. Das macht Ihr Programm langsamer oder sogar langsam.

Typische Kandidaten:

  • Reguläre Ausdrücke

  • Formatter wie DateTimeFormatter, NumberFormatter, SimpleDateFormat

  • Collections

Ineffiziente Anwendung eines regulären Ausdrucks 
for (String str : strings) {
    if (str.matches("^[a-z]+$")) { (1)
        // do something
    }
}
1 str.matches("^[a-z]+$") ist gleichbedeutend mit Pattern.compile("^[a-z]+$").matcher(str).matches(), d.h. bei jedem Durchgang wird derselbe reguläre Ausdruck erneut kompiliert. Das ist eine Verschwendung von CPU-Zyklen.

Erstellen Sie reguläre Ausdrücke oder Formatter so wenig wie möglich, indem Sie sie aus Schleifen herausnehmen oder, noch besser, in statische Felder umwandeln.

Effiziente Anwendung eines regulären Ausdrucks 
public class Example {
    private static final Pattern STR_PATTERN = Pattern.compile("^[a-z]+$"); (1)

    public void doSomething(List<String> strings) {
        for (String str : strings) {
            if (STR_PATTERN.matcher(str).matches()) { (1)
                // do something
            }
        }
    }
}
1 Das ist auch in Servlet-Umgebungen sicher anzuwenden, weil Pattern thread-safe ist. Schauen Sie ins Javadoc der entsprechenden Klassen, wenn Sie unsicher sind, ob eine Klasse thread-safe ist.

Auch mit Collections lässt sich viel Schindluder anstellen. Ein klassisches Eigentor ist, sie immer wieder neu zu erstellen:

Ineffiziente Verwendung einer Collection 
for (String str : strings) {
    List<SomeEntity> myList = new ArrayList<SomeEntity>();
    // do something with myList
}

Verwenden Sie stattdessen die Collection wieder, indem Sie die Methode clear() anwenden.

Effiziente Verwendung einer Collection 
List<SomeEntity> myList = new ArrayList<SomeEntity>(); (1)
for (String str : strings) {
    myList.clear(); (2)
    // do something with myList
}
1 Die Liste wird nur noch einmal ausserhalb der Schleife erstellt.
2 clear() leert die Liste und sie lässt sich wiederverwenden, als wäre sie "neu".
Verwandeln Sie Collections, in die Sie schreiben, nur in Ausnahmefällen in statische Felder — selbst wenn sie thread-safe sind. Die Synchronisation mehrerer schreibender Threads ist aufwendiger und fehleranfälliger als die Erstellung mehrerer Collections.

Angemessene Verwendung von Collections (L1)

Wenn Sie Objekte in Collections (Array, List, …) ablegen, achten Sie darauf, die geeignete Collection zu wählen. Die Auswahl ist nämlich gross und Sie können sich dadurch eine Menge Arbeit sparen. Ein SortedSet oder eine SortedMap sind zum Beispiel implizit sortiert, bei einer ArrayList müssen Sie sich selbst darum kümmern.

  • Wenn Sie spezielle Anforderungen haben, studieren Sie die Übersicht über das Java Collections Framework, um eine passende Collection zu finden.

  • Verwenden Sie ArrayList anstelle von Arrays. ArrayList ist typsicher während der Übersetzung, d.h. der Compiler hält Sie davon ab, ungültige Datentypen in einer ArrayList zu speichern. Arrays sind dagegen nur zur Laufzeit typsicher. ArrayList ist ausserdem flexibler bezüglich Iteration, unterscheidet zwischen belegten Elementen und seiner tatsächlichen Länge und seine Länge kann dynamisch verändert werden.

  • Die Hilfsklasse Collections bietet eine grosse Auswahl nützlicher Methoden an, u.a. zum Kopieren und Sortieren von Collections. Wenn Sie Collections mit keinem oder nur einem Element benötigen, verwenden Sie Collections.emptyList() und Konsorten beziehungsweise Collections.singletonList(T o) und so weiter. Diese sind effizienter, als wenn Sie selber die entsprechenden Collections erstellen.

  • Verzichten Sie auf java.util.Vector.

Studieren Sie das Javadoc der verschiedenen Collections und insbesondere deren Konstruktoren. Eine LinkedBlockingQueue kann man zum Beispiel mit den richtigen Konstruktor-Argumenten in ihrer Grösse begrenzen oder aus einer LinkedHashMap durch überschreiben einer Methode einen Cache machen.

Favor functional over imperative style (L4)

Neu!

Java erlaubt sowohl eine imperative als auch funktionale Arbeitsweise. Es lohnt sich zu wissen, wann man das eine oder das andere benutzt. Bei Listen lohnt sich meistens das funktionale vorgehen. Wie in folgendem Beispiel:

                }

        }
